# Calendar-integration audit — 2026-07-19

Scope: the true **calendar** providers — Google Calendar (`google`), Outlook /
Microsoft 365 (`outlook`), Apple / CalDAV (`apple`) — plus the shared machinery
they depend on (`graph.ts`, `ical.ts`, `tz.ts`, `time.ts`, `http.ts`, `errors.ts`)
and their webhook verifiers. `microsoft_bookings` was reviewed incidentally
because it reuses the Graph helpers.

Every finding below was checked against the current provider specs:

- Google Calendar API v3 — <https://developers.google.com/workspace/calendar>
- Microsoft Graph v1.0 (events, calendarView, subscriptions) — <https://learn.microsoft.com/graph>
- CalDAV **RFC 4791** and iCalendar **RFC 5545**

Baseline before the audit: 354 tests green, `tsc --noEmit` clean, `eslint` clean.
After: 361 tests green (7 added), typecheck + lint clean.

## Bugs fixed

### 1. `412 Precondition Failed` was retryable — defeating CalDAV lost-update protection

**Severity: high (silent data loss under `withRetry`).**

The Apple/CalDAV adapter guards writes with ETags: `updateBooking` sends
`If-Match: <etag>` and `createBooking` sends `If-None-Match: *`, so a concurrent
edit or a UID collision fails the write instead of clobbering. Both failures come
back as HTTP **412**. But `codeForStatus` had no `412` case, so it fell through to
**`UPSTREAM`** — which `isRetryable()` treats as retryable. Under `withRetry`, an
`updateBooking` that hit a 412 would be **re-run**: the retry re-`GET`s a fresh
ETag and re-`PUT`s, **silently overwriting the concurrent edit** the guard existed
to protect. The whole optimistic-concurrency story was undone by the retry
wrapper.

**Fix:** `codeForStatus(412) → 'CONFLICT'` (`src/errors.ts`). `CONFLICT` is
non-retryable, so `withRetry` now surfaces the conflict to the caller instead of
clobbering. Regression: `test/http.test.ts` (mapping) and
`test/adapters/audit-fixes.test.ts` (Apple `updateBooking` 412 → `CONFLICT`).

### 2. iCalendar `DURATION` week form (`P1W`) collapsed to a zero-length range

**Severity: medium (wrong event end for a valid input).**

RFC 5545 `dur-value` allows a standalone week form (`dur-week = 1*DIGIT "W"`),
e.g. `P1W` = 7 days. `ical.applyDuration`'s regex only matched days/hours/minutes/
seconds, so a `VEVENT` sized by `DURATION:P1W` (no `DTEND`) failed to parse the
duration and fell through to the "no duration" default — producing
`end === start`, a zero-length booking. This is the same zero-length-range class
the previous audit hardened Acuity/Vagaro/Phorest/Square against.

**Fix:** `applyDuration` now handles the week form (`src/ical.ts`). Regression:
`test/ical.test.ts` (`P1W` → 7-day span).

### 3. Google `updateBooking` dropped non-`cancelled` status changes

**Severity: low (missing behavior / cross-adapter inconsistency).**

Google models `tentative`/`confirmed`/`cancelled` as the writable event `status`
field (verified against the v3 Events reference). The adapter only forwarded
`status: 'cancelled'`; a `pending` or `confirmed` update was silently dropped.
Outlook (`showAs`) and Apple (iCal `STATUS`) both map status on update, so Google
was the odd one out.

**Fix:** `updateBooking` maps `pending → tentative`, `confirmed`/`completed →
confirmed`, `cancelled → cancelled`, and leaves the field untouched for statuses
with no Google form (`src/adapters/google.ts`). Regression:
`test/adapters/audit-fixes.test.ts`.

## Known limitations (documented, behavior unchanged)

These are correctness gaps or missing features where a runtime change was either
out of scope or couldn't be verified against the live provider. They are now
called out in the provider docs so they don't surprise consumers.

### A. Apple/CalDAV `listBookings` recurring expansion — FIXED (follow-up pass)

Originally a plain `calendar-query` time-range `REPORT` returned the **unexpanded
master** `VEVENT` (RFC 4791 §7.6/§7.8.1) — original `DTSTART`/`DTEND` (possibly
*before* the window) plus `RRULE` — so a repeating series yielded one booking at
the wrong time, unlike Google/Outlook.

**Fix (follow-up pass):** the `calendar-query` now requests `<C:expand start end/>`
(RFC 4791 §9.6.5), so the server returns one concrete `VEVENT` per in-window
occurrence and the adapter maps each to its own booking with the right time.
Regression in `test/adapters/apple.test.ts`. Residual caveats (documented in
[docs/providers/apple.md](../providers/apple.md)): iCloud's server-side expand is
[occasionally inconsistent](https://developer.apple.com/forums/thread/94363), and
expanded instances of one series share a DAV resource → share a booking `id` (use
`raw`'s `RECURRENCE-ID` to disambiguate). No client-side `RRULE` fallback.

### B. Outlook `cancelBooking({ notify | reason })` uses the organizer-only `/cancel`

Graph's `POST /events/{id}/cancel` is available **only to the meeting organizer**
and returns `400` ("You need to be an organizer to cancel a meeting") otherwise;
its behavior on a personal, attendee-less event is undocumented. The adapter routes
to `/cancel` whenever `notify`/`reason` is set. A no-`notify` cancel uses a plain
`DELETE`, which always works. Documented in
[docs/providers/outlook.md](../providers/outlook.md). A future improvement could
fall back to `DELETE` on a 400 from `/cancel`, but that masks legitimate 400s and
wasn't verifiable against live Graph here.

### C. Google notify-on-create — FIXED (follow-up pass); Outlook still automatic

Google sends no invitations unless the `sendUpdates` query param is set. A
follow-up pass added a canonical **`notify?: boolean`** to `CreateBookingInput`/
`UpdateBookingInput` and wired it into Google's create/update (`notify: true` →
`sendUpdates=all`, `false` → `none`, omitted → Google default). Regression in
`test/adapters/audit-fixes.test.ts`. Outlook/Graph sends meeting invitations
automatically when an event has attendees, so no separate toggle is wired there;
other providers ignore `notify` unless they support it.

## Verified correct (no change needed)

- **Google** `events.list`: `orderBy=startTime` correctly pairs with
  `singleEvents=true`; `timeMin`/`timeMax` overlap semantics are as intended;
  all-day events use exclusive `end.date` → midnight-UTC instants. No breaking
  2025/2026 API changes affect basic CRUD or watch channels.
- **Outlook** `calendarView` with `$orderby=start/dateTime` + `$top` + full
  `@odata.nextLink` paging is supported; the `Prefer: outlook.timezone="UTC"`
  header shapes only the response TZ (the query window uses the offset in the
  literals), which is exactly how the adapter uses it; `graphToInstant` correctly
  resolves offset-bearing, naive-UTC, and Windows/IANA-zoned datetimes.
- **Apple** `updateBooking` in-place `patchICS` preserves `RRULE`/`LOCATION`/
  `DESCRIPTION`/extra attendees/`VALARM`/`VTIMEZONE`; href-based ids round-trip;
  `If-Match`/`If-None-Match` guards are correct (and now surface as `CONFLICT`).
- **Webhooks**: Google channel-token compare, Graph `validationToken` handshake,
  and `clientState` compare are all constant-time and correctly reject the
  misconfigured/empty cases.
